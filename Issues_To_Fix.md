# Issues & Bugs to Fix

> Generated by comprehensive code review of the Arama project (Next.js 16.2.9, Drizzle ORM, better-sqlite3)

## 🔴 CRITICAL BUGS

### C1. Forgot Password: Generated Token Not Passed to Email Function (BROKEN FEATURE)

- **File:** `src/app/api/auth/forgot-password/route.ts` (Lines 50-62)
- **Problem:** `crypto.randomUUID()` generates a token on line 51, it's stored in `passwordResetTokens` table on line 54-59, BUT the `sendPasswordResetEmail()` call on line 62 does NOT receive this token. Meanwhile, `src/lib/email/index.ts` (line 73-105) `sendPasswordResetEmail` generates its OWN token and creates a NEW separate DB record. The email contains the token from `email/index.ts`, but the URL in the email won't match the token stored by the route.
- **Result:** Password reset is completely broken. Users never receive a working reset link.
- **Fix:** Pass the generated `token` to `sendPasswordResetEmail(user.id, email, token)` and update the email function signature to accept and use the pre-generated token instead of creating its own.

### C2. `useEffect` Duplicate Listener Registration (Potential Memory Leak / Re-render Loop)

- **File:** `src/hooks/use-toast.ts` (Lines 176-184)
- **Problem:** `React.useEffect` depends on `[state]`. Every time the toast state changes, the effect re-runs and pushes a new `setState` into the `listeners` array WITHOUT removing the old one. This causes:
  1. Duplicate listener registration on every toast update
  2. Multiple `setState` calls on every dispatch, causing unnecessary re-renders
  3. Potential infinite re-render loop if `setState` triggers state changes
- **Fix:** Change dependency to `[]` so the effect only runs once on mount, registering the listener only once. The cleanup function (lines 178-182) already handles proper removal on unmount.

### C3. `isStreaming` Stale Closure in useChat (`send` callback)

- **File:** `src/hooks/useChat.ts` (Lines 66-163, particularly line 78 `if (isStreaming)`)
- **Problem:** The `send` callback has `[conversationId, isStreaming]` in its dependency array. The `isStreaming` guard on line 78 uses a stale closure value. When `isStreaming` changes, the entire `send` function is recreated, but any in-flight calls captured the old reference.
- **Fix:** Use a `useRef<boolean>` for the streaming guard (e.g., `isStreamingRef`), check `isStreamingRef.current` instead of `isStreaming`, and remove `isStreaming` from the dependency array.

### C4. `onCreate` Called Without Required `title` Argument

- **File:** `src/components/chat/ConversationList.tsx` (Lines 50-55) calls `onCreate()` without arguments.
- **File:** `src/hooks/useChat.ts` (Line 27) `create(title: string)` expects a `title` argument.
- **Result:** `title` will be `undefined`, causing conversations to be created with `title || "مکالمه جدید"` fallback (which works but is unintended behavior). More importantly, if `onCreate` is supposed to trigger creation, users will never get to name their conversations.
- **Fix:** Either pass a default title or implement a dialog for conversation naming before calling `onCreate`.

### C5. API Routes NOT Protected by Middleware (Authentication Gap)

- **File:** `src/proxy.ts` (Lines 35-37)
- **Problem:** The middleware `config.matcher` pattern `"/((?!api|_next/static|_next/image|favicon.ico).*)"` EXCLUDES `/api/*` routes from middleware execution. This means the auth check in `proxy.ts` (lines 24-30) never runs for API routes. Individual API routes rely on `requireAuth()` calls, but any newly created API route that forgets to call `requireAuth()` will be publicly accessible.
- **Fix:** Do NOT exclude API routes from middleware, OR add explicit authentication check logic inside the middleware for API routes. At minimum, document the risk clearly.

### C6. Dockerfile References Prisma but Project Uses Drizzle ORM

- **File:** `Dockerfile` (Lines 27, 46)
- **Problem:** Line 27: `RUN npx prisma generate` - Prisma is not installed (it's not in `package.json`). Line 46: `COPY --from=builder /app/node_modules/.prisma ./node_modules/.prisma` - copies non-existent directory.
- **Fix:** Remove all Prisma references from the Dockerfile. Drizzle ORM doesn't need a generation step at build time (schema is defined in TypeScript).

## 🟠 BUGS

### B1. `user.roles` Might Be `undefined` - `.some()` Will Crash

- **File:** `src/app/(admin)/admin/layout.tsx` (Line 16)
- **Problem:** `requirePermission()` returns user with a `roles` property that may be `undefined` (the `requirePermission` function in `auth-helpers.ts` line 61-66 returns `roles` from `getUserRoles` but the function's return type doesn't guarantee it). Calling `user.roles.some(...)` on undefined throws a runtime error.
- **Fix:** Use optional chaining: `const hasAdminRole = (user.roles ?? []).some(...)`

### B2. Unused Imports in Multiple Files

- **Files:**
  - `src/lib/audit.ts` (Line 4): `headers` from `next/headers` imported but never used
  - `src/app/api/auth/register/route.ts` (Lines 10-11): `asc, desc` from `drizzle-orm` unused
  - `src/app/api/auth/forgot-password/route.ts` (Lines 7, 11): `asc, desc` from `drizzle-orm` unused; `randomUUID` imported but line 51 uses `crypto.randomUUID()` instead
- **Fix:** Remove unused imports.

### B3. Inconsistent ID Types: `number` vs `string` (UUIDs)

- **File:** `src/hooks/useChat.ts`
  - Line 39: `remove` expects `id: number`
  - Line 47: `useMessages(conversationId: number | null)`
  - Lines 57-60: Message mapping renames `id` as `String(m.id)` - suggests backend returns `number` IDs
- **Context:** Database schema (`src/db/schema.ts`) defines all IDs as `text("id")` with `crypto.randomUUID()`. Frontend code uses `number` IDs, backend generates string UUIDs.
- **Fix:** Change all ID types in hooks to `string` and ensure API responses return string IDs. The mapping on lines 57-60 suggests this mismatch needs a systematic fix.

### B4. Rate Limiter is In-Memory Only (Not Production-Ready)

- **File:** `src/lib/rate-limit.ts` (Line 5)
- **Problem:** Uses a `Map<string, { count: number; timestamp: number }>()` in memory. Rate limits are:
  1. Lost on server restart (Vercel/Edge functions, serverless)
  2. Not shared across multiple instances
  3. Can grow unbounded (no Map cleanup for expired entries)
- **Fix:** Use a persistent store (Redis via Upstash, or database-backed rate limiting).

### B5. `getAuditLogs` Returns Empty Array Always (Stub Function)

- **File:** `src/lib/audit.ts` (Lines 41-57)
- **Problem:** The function always returns `[]` with a comment "Implementation would go here." Any code calling this in production gets no data silently.
- **Fix:** Implement the query or throw a "Not implemented" error.

### B6. Potential Serverless Module Import Issues (better-sqlite3)

- **File:** `src/lib/db.ts` (Lines 1-50)
- **Problem:** `better-sqlite3` is a native Node.js module. Importing it at module scope means even routes that don't use the database will cause failures in Edge Runtime. The `fs.existsSync`/`fs.mkdirSync` calls on lines 41-43 are synchronous filesystem operations that block the event loop at import time.
- **Fix:** 
  1. Add `better-sqlite3` to `serverExternalPackages` in next.config.ts (already done, line 9)
  2. Consider lazy-loading the database connection
  3. Move sync filesystem operations to async

### B7. Unnecessary `.env` File Override Pattern

- **File:** `src/lib/db.ts` (Lines 30-33)
- **Problem:** Calls `dotenv.config()` twice. `dotenv` is for development only; in production, environment variables should be set natively. The custom `.env.local`/`.env.production`/etc. loading on lines 13-27 adds complexity.
- **Fix:** Simplify - use Next.js built-in `.env` file loading. Remove `dotenv` usage from production code.

### B8. Ambiguous Explicit Any Types

- **File:** `src/lib/auth.config.ts`
  - Line 94: `jwt({ token, user }: { token: any; user: any })`
  - Line 103: `session({ session, token }: { session: any; token: any })`
  - Line 33: `authorize(credentials: any)`
- **Fix:** Use proper NextAuth type generics (`JWT`, `Session`, `User`) instead of `any`.

### B9. `randomUUID` Import Inconsistency

- **File:** `src/app/api/auth/forgot-password/route.ts` (Line 51)
- **Problem:** `randomUUID` is imported from `crypto` (line 11), but line 51 uses `crypto.randomUUID()` instead of the imported `randomUUID`. This is inconsistent and confusing.
- **Fix:** Use the imported `randomUUID` directly: `const token = randomUUID();`

## 🟡 POTENTIAL BUGS / CODE QUALITY

### Q1. No Input Validation for `title` in Conversation Create

- **File:** `src/app/api/chat/conversations/route.ts` (Line 30)
- **Problem:** `const { title } = await _request.json();` - No validation on the title field. Could be malicious HTML/script content, excessive length, etc.
- **Fix:** Add Zod validation for the request body.

### Q2. Payment Service Creates Subscription BEFORE Payment Confirmed

- **File:** `src/lib/services/payment/index.ts` (Lines 47-60)
- **Problem:** A `subscriptions` record with status "PENDING" is created immediately upon requesting payment. If the user never completes payment, there's a dangling PENDING subscription with no cleanup mechanism.
- **Fix:** 
  1. Add cleanup cron job for expired PENDING subscriptions
  2. Or create subscription record only after payment verification

### Q3. Payment Refund Doesn't Actually Refund via Gateway

- **File:** `src/lib/services/payment/index.ts` (Lines 197-221)
- **Problem:** `refundPayment()` only updates the local database status to "REFUNDED". It never actually calls the payment gateway API to process the refund. The comment on lines 195-196 acknowledges this.
- **Fix:** Integrate actual refund API calls for each gateway.

### Q4. `logAudit` Silently Fails (Error Suppression)

- **File:** `src/lib/audit.ts` (Lines 24-38)
- **Problem:** The `catch` block on line 35-38 only logs to console and swallows the error. This is intentional for audit logging, but means failed audit writes are invisible in production monitoring.
- **Fix:** Consider using a dedicated logging service or at least emitting a metric on audit failure.

### Q5. No CSRF Protection on Cookie-Based Auth

- **File:** `src/lib/auth.config.ts`
- **Problem:** The app uses JWT sessions (line 24: `strategy: "jwt"`), which rely on cookies. There's no CSRF token validation on state-changing requests (POST, PUT, DELETE). An attacker could forge cross-site requests.
- **Fix:** Either:
  1. Add CSRF token validation middleware
  2. Use NextAuth's built-in CSRF protection (if available for JWT strategy)
  3. Use `SameSite: Strict` or `Lax` cookies (Next.js defaults)
  4. Implement double-submit cookie pattern

### Q6. Missing IP Address Validation / Spoofing Risk

- **File:** `src/lib/audit.ts` (Lines 59-79)
- **Problem:** `getClientInfo()` trusts `x-forwarded-for`, `x-real-ip`, `x-client-ip`, and `cf-connecting-ip` headers without validation. An attacker can spoof their IP by sending arbitrary header values. This affects audit logs and rate limiting.
- **Fix:** Only trust these headers from known reverse proxies. Use `request?.ip` (NextRequest built-in) as primary source.

### Q7. Token Verification Tokens Not Cleaned Up

- **File:** `src/db/schema.ts` (Lines 81-108)
- **Problem:** `emailVerificationTokens` and `passwordResetTokens` tables have no automatic cleanup for expired tokens. They use `usedAt` field but don't purge expired rows, leading to table bloat over time.
- **Fix:** Add a cleanup job (Node-cron is already in dependencies) to periodically remove expired tokens.

### Q8. `trustHost: true` Disables Host Header Validation

- **File:** `src/lib/auth.config.ts` (Line 116)
- **Problem:** `trustHost: true` tells NextAuth to skip host header validation against `NEXTAUTH_URL`. This can open up host header injection vulnerabilities if not carefully managed.
- **Fix:** Remove `trustHost: true` and properly configure `NEXTAUTH_URL`. Only use `trustHost: true` behind a known proxy, and combine with a list of trusted hosts.

## 🔴 SECURITY & PACKAGE RISKS

### S1. `next-auth@5.0.0-beta.25` - Beta/Pre-release Authentication

- **Package:** `next-auth@5.0.0-beta.25`
- **Risk:** HIGH. Beta software may contain undiscovered security vulnerabilities, breaking changes, or instabilities. The API surface may change without notice between beta versions. Critical auth bugs in production could compromise all user accounts.
- **Fix:** 
  1. Monitor the Auth.js repository for GA release
  2. Pin to an exact beta version commit hash
  3. Consider using the stable v4 (`next-auth@4.24.14`) as a fallback
  4. Regularly update and test the beta version

### S2. `better-sqlite3` in Production Web Application

- **Package:** `better-sqlite3@12.11.1`
- **Risk:** HIGH. SQLite is not designed for concurrent production web workloads:
  1. **No concurrent writes** - Multiple server instances share the same file; write conflicts
  2. **No horizontal scaling** - Cannot add more server instances
  3. **File-based locking** - May cause `SQLITE_BUSY` errors under load
  4. **Not suitable for Vercel/Serverless** - File doesn't persist across function invocations
  5. **docker-compose.yml already references PostgreSQL** (`DATABASE_URL: postgresql://...`) indicating the intended target is actually PostgreSQL, but code uses better-sqlite3
- **Fix:** Either:
  1. Switch to the PostgreSQL driver for Drizzle ORM (`drizzle-orm/pg-core` with `pg` or `@neondatabase/serverless`)
  2. Or keep SQLite for development/local and use PostgreSQL for production

### S3. Hardcoded Secrets in `docker-compose.yml`

- **File:** `docker-compose.yml`
  - Line 19: `NEXTAUTH_SECRET: development-secret`
  - Line 26: `CRON_AUTH_TOKEN: dev-cron-token`
  - Lines 36-38: `POSTGRES_PASSWORD: arama`
- **Risk:** These defaults could be deployed to production unchanged. Anyone who has access to the docker-compose file (it's in the repo) knows the auth secret and DB password.
- **Fix:** 
  1. Remove hardcoded secrets from docker-compose.yml
  2. Use environment variable references: `NEXTAUTH_SECRET: ${NEXTAUTH_SECRET}`
  3. Generate strong random secrets for production

### S4. Database Password in Environment Variable Visible in Commands

- **File:** `docker-compose.yml` (Lines 16-17)
- **Risk:** Database credentials `postgresql://arama:arama@db:5432/arama` are exposed in plain text in the compose file and in any logs showing environment.
- **Fix:** Use Docker secrets or environment variable substitution: `DATABASE_URL: ${DATABASE_URL}`

### S5. REST API Endpoints Return Detailed Error Messages in Production

- **Files:** Multiple API routes (e.g., `register/route.ts` line 123, `forgot-password/route.ts` line 80)
- **Problem:** `details: err instanceof Error ? err.message : "خطای ناشناخته"` - Error messages from caught exceptions are returned to the client in 500 responses. In production, these can leak:
  1. Internal server paths
  2. Database error details (table names, column info)
  3. Stack traces
- **Fix:** In production, return a generic error message and log the detailed error server-side.

### S6. Rate Limiting Bypass: IP + Email Key Can Be Manipulated

- **File:** `src/lib/rate-limit.ts` (Line 14)
- **Problem:** The rate limit key is `${ip}:${email}`. An attacker can:
  1. Use different emails for each attempt to bypass IP-based limits
  2. Use IP spoofing (see Q6) to bypass per-IP limits
- **Fix:** Implement rate limiting at multiple levels (per-IP, per-email, global) and validate IP headers.

### S7. `react-redux@8.1.3` + `redux@4.2.1` vs React 19 Compatibility

- **Package:** `react-redux@8.1.3`, `redux@4.2.1`
- **Risk:** React-Redux 8.x targets React 18. The project uses React 19.2.7. While React-Redux 8 may work with React 18's legacy API, React 19 may have breaking changes that affect Redux compatibility. The project also uses `zustand@5.0.14` - maintaining two state management libraries (Redux + Zustand) adds unnecessary complexity.
- **Fix:** 
  1. Upgrade to `react-redux@9.x` (React 19 compatible)
  2. Or migrate fully to Zustand (already in use)
  3. Or remove Redux/Zustand and standardize on one

### S8. No Input Rate Limiting on Registration/Contact Endpoints

- **Risk:** Registration endpoint has no rate limiting. An attacker could:
  1. Script account creation for spam/dos
  2. Perform email enumeration (even with generic response messages, timing differences may leak info)
- **Fix:** Add rate limiting to registration endpoint and other public POST endpoints.

### S9. `helmet@8.2.0` Installed But Not Used

- **Package:** `helmet@8.2.0` (in `package.json` line 63)
- **Problem:** Helmet is installed in the Node.js server dependencies but:
  1. Next.js doesn't use Express; Helmet is primarily for Express.js
  2. Security headers are already set via `next.config.ts` (lines 18-31)
- **Fix:** Remove `helmet` from dependencies as it provides no value in this context.

### S10. Weak Demo User Credentials Hardcoded in Seed Script

- **File:** `drizzle/seed.ts`
  - Line 367: `const demoUserEmail = "sara_mohammadi@gmail.com";`
  - Line 368: `const demoUserPassword = "sara_mohammadi";` (password = email username)
  - Line 737: `const superAdminPassword = process.env.SUPER_ADMIN_PASSWORD || "Admin123!@#";` (fallback hardcoded)
- **Problem:** 
  1. Demo user password is the same as the email username part (`sara_mohammadi`), making it trivially guessable
  2. If `SUPER_ADMIN_PASSWORD` env var is not set in production, it defaults to `Admin123!@#`
  3. `Admin123!@#` follows a predictable pattern (common word + 123 + special chars)
  4. The super admin credentials are printed to stdout on line 785 in bold `=== Seeding Complete ===` output
- **Fix:** 
  1. Generate random passwords for seed data, or use strong passphrases
  2. Remove the fallback default for `SUPER_ADMIN_PASSWORD` - fail if not set
  3. Never print credentials to console in production

### S11. `dotenv@16.4.5` in Production Dependencies

- **Package:** `dotenv@16.4.5`
- **Problem:** `dotenv` is listed in `dependencies` (runtime) but should be in `devDependencies`. Next.js has built-in `.env` file loading. The manual `dotenv.config()` calls in `src/lib/db.ts` override Next.js's environment loading.
- **Fix:** Move `dotenv` to `devDependencies` or remove it entirely and rely on Next.js built-in env loading.

## 🟠 INFRASTRUCTURE & DEPLOYMENT

### I1. Dockerfile Build Failure (Prisma + Missing Lockfile)

- **File:** `Dockerfile`
  - Line 16: Missing `.npmrc` for private registries
  - Line 14: Uses `npm ci` if lockfile exists, but uses `yarn global add pnpm` if only pnpm-lock exists - this is a broken fallback
  - Line 27: `npx prisma generate` - will fail since Prisma is not a dependency
  - Line 34: `yarn build` - but above uses `npm`, `yarn`, or `pnpm` inconsistently
- **Fix:** Rewrite Dockerfile to be consistent with the actual project setup (npm lockfile exists: `package-lock.json`).

### I2. Missing `init.sql` for Database Initialization

- **File:** `docker-compose.yml` (Line 41): `- ./init.sql:/docker-entrypoint-initdb.d/init.sql`
- **Problem:** `init.sql` file does not exist in the project root. Docker will fail to start if this file is expected.
- **Fix:** Either create the `init.sql` file or remove the volume mount.

### I3. Cron Service Duplicates Full Application Container

- **File:** `docker-compose.yml` (Lines 59-79)
- **Problem:** The `cron` service builds the same image as the `app` service and runs `node server.js`. This starts a full Next.js server just for cron jobs, wasting resources.
- **Fix:** Create a dedicated lightweight cron container or use an external cron scheduler.

### I4. Missing `CSP` (Content Security Policy) Header

- **File:** `next.config.ts` (Lines 18-31)
- **Problem:** Security headers are set but Content-Security-Policy is missing. Without CSP, XSS attacks are easier to execute.
- **Fix:** Add a strong CSP header tailored to the application's needs (script-src, style-src, etc.).

### I5. `NEXTAUTH_URL` Set to `http://localhost:8080` in Docker

- **File:** `docker-compose.yml` (Line 18)
- **Problem:** Inside Docker, the app listens on port 8080, which is correct. But in production, this must be the public-facing URL (e.g., `https://arama.app`). If misconfigured, OAuth callbacks will break and redirect to localhost.
- **Fix:** Make `NEXTAUTH_URL` configurable via environment variable override in production deployments.
